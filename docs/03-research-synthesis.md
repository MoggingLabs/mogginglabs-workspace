# Building a Cross-Platform AI-Agent CLI Organizer — Research Synthesis & Build Plan

**Audience:** technical founder. **Goal:** ship a custom, Windows+macOS-parity version of BridgeMind's *BridgeSpace* that organizes multiple first-party coding-agent CLIs (Claude Code, Codex, Gemini CLI, Aider, OpenCode, …) in persistent workspaces — a neutral rival to Warp, tmux, and cmux.

**TL;DR of the recommendation:**
- **Positioning:** be the *neutral organizer of CLIs* (never broker provider auth) with **true Windows+macOS parity and rock-solid terminal rendering** — precisely where the best rivals are weak (cmux/Conductor are macOS-only; BridgeSpace does Windows but on Tauri with a documented multi-month history of terminal-rendering/freeze bugs; Warp is closed and is pivoting to push *its own* agents).
- **Stack:** **Electron + xterm.js (WebGL addon) + node-pty**, with a persistent PTY-host process and a platform-agnostic TypeScript "engine" package. This gives one Chromium rendering engine on both OSes and real PTYs (ConPTY on Windows, forkpty on macOS) from one codebase. This is the exact stack VS Code's terminal is built on.
- **MVP:** multi-pane terminal that hosts real agent CLIs, persistent workspaces, OSC-based agent-state detection, Warp-style command blocks. Defer swarm/memory/editor/voice.

---

## 1. What BridgeSpace actually is, and what we must match or beat

**What it is.** BridgeSpace (BridgeMind, currently v3.2.2, June 2026) is a native cross-platform desktop app — **Tauri (Rust core + system WebView2/WKWebView) + xterm.js terminals** — that markets itself as an "Agentic Development Environment." Concretely it is a **Warp-style multi-pane terminal that hosts real, terminal-based coding-agent CLIs as OS subprocesses** in 1–16 panes, wrapped with orchestration and IDE-like surfaces:

- **Terminal core:** split grids (templates for 1/2/4/6/8/10/12/14/16 panes), Warp-style *command blocks* (per-command collapsible sections, exit-code color, timestamps, Cmd/Ctrl+F search), GPU rendering, 25+ themes.
- **Agent roster (real CLIs, self-authenticated):** Claude Code (deepest integration — installs `~/.claude` hooks, `SessionStart` registration, `--effort max` default, multi-account profiles), OpenAI Codex, Gemini CLI, Cursor agent, OpenCode, Grok Build.
- **Orchestration:** **BridgeSwarm** (roles Coordinator/Builder/Scout/Reviewer, *exclusive file ownership* to avoid merge conflicts, shared mailbox, Reviewer quality gate; "up to 16 agents," though its own FAQ says 10 panes / 3–5 agents per swarm — a documented inconsistency).
- **Coordination surfaces:** Kanban board (Todo/In Progress/In Review/Complete) that auto-launches an agent into a pane with context; integrated code editor + file tree; Git sidebar with status/diff/commit/worktree and a **read-only "pre-ship diff review" that redacts secrets and resists prompt injection**; built-in browser; voice "Bridge Assistant."
- **Memory:** **BridgeMemory**, a local-first markdown knowledge graph in `.bridgememory/`, shared to agents via ~12 MCP tools and `[[wikilinks]]`.
- **Auth model (important):** **it never brokers provider auth.** "Connect Accounts" links *your* Claude/Codex accounts and drives the real CLIs signed in under them; a separate "API Keys" section holds BYOK. BridgeMind's own paid "credits" power **only** first-party features (Bridge Assistant, voice). Pricing: Basic $16, Pro $40, Ultra $80/mo (annual), credits by tier.
- **Reality check:** closed source (the `bridge-mind` GitHub org has only 5 small unrelated repos), modest traction (~17 Product Hunt upvotes), and a changelog dominated by **multi-agent terminal-rendering/freeze fixes** — its clearest technical weak spot.

**What our custom version MUST MATCH (table stakes):**
1. Host the same CLI roster as **real PTY subprocesses** in resizable panes — TUIs (alt-screen, raw mode, color) must work perfectly.
2. **Never broker provider auth.** Users run their own CLIs under their own subscriptions/keys; we only orchestrate which CLI/profile/env is active.
3. Persistent, restorable **workspaces** (project dir + pane layout + cwd + which agent).
4. **Warp-style command blocks** with exit codes; per-pane git awareness; agent-state awareness (who needs attention).
5. Shell-config fidelity (`.zshrc`/`.bashrc`/PATH preserved), a `yourapp .` launcher, themes, GPU rendering.

**What we must BEAT (our wedge, see §3):**
- **True Windows+macOS parity with best-in-class terminal reliability** under many parallel agents — attack their documented rendering/freeze weakness head-on.
- **Neutrality + scriptability** (tmux-grade control API) vs. tools that push their own agent.
- **Leanness and fast startup.**

**De-scope for MVP:** BridgeSwarm, BridgeMemory/MCP graph, integrated editor, browser, voice. These are v1+ differentiators, not the core organizer.

## 3. Gaps and our wedge

**Primary wedge — the Windows+macOS parity gap.** The best-*designed* tools in this category deliberately skip Windows:
- **cmux** (the category's fastest-growing, ~23k stars, best OSC-based agent-awareness) is **macOS-only** (Swift/libghostty). Windows/Linux exist only as unofficial forks.
- **Conductor** (the cleanest commercial parallel-agents UX) is **macOS-only**.
- **Sculptor** is Mac+Linux (no Windows).
- **tmux** isn't native Windows (WSL only).
- The two rivals that *do* ship Windows are compromised for our target user: **BridgeSpace** does it on Tauri with two divergent WebView engines and a **documented multi-month history of terminal-rendering/freeze bugs**; **Warp** ships Windows but is **closed and pivoting to its own agent product**, so it's not a neutral organizer.

→ **Our wedge:** *the most reliable, native-feeling, identical-on-both-OSes organizer of first-party agent CLIs, with best-in-class terminal rendering.* We win the large, underserved population of Windows-primary and mixed Windows/macOS teams (enterprise, fintech, gamedev, .NET/Unity shops) that macOS-only tools ignore — and we beat BridgeSpace on the exact axis (rendering reliability) where its own changelog shows it bleeding.

**Secondary openings:**
1. **Neutrality.** Warp and coder/mux increasingly push *their own* agent/model loop. We stay strictly agent-agnostic and **never broker provider auth** — a trust and simplicity advantage, and a legal/ToS de-risker.
2. **Scriptability as a first-class feature.** tmux proves developers love a control API; the GUI rivals under-invest here. A documented socket + `yourapp send/send-key/list` API (à la cmux/tmux) makes us automatable and CI-friendly.
3. **Reliability at high pane counts.** "16 agents streaming at once without freezing" is a concrete, demoable win over BridgeSpace.
4. **Licensing trust.** The strongest OSS rivals are **AGPL-3.0 (coder/mux, claude-squad)** or **GPL-3.0 (cmux)** — copyleft that scares commercial adopters. A permissive/source-available posture is a differentiator for teams that can't touch AGPL.
5. **Lean footprint & fast cold start** vs. a feature-maximalist competitor.

## 4. Recommended MVP feature set (organizer-of-CLIs framing)

**Must-have (MVP / v0 — "a reliable multi-pane host for agent CLIs on Windows+macOS"):**
1. **One codebase, real PTYs on both OSes** — ConPTY (Windows) + forkpty (macOS) via node-pty. Non-negotiable; it's the whole product.
2. **Multi-pane terminal** — split + grid templates (1/2/4/quad/6…), drag-resize, xterm.js + WebGL rendering, copy/paste, find, per-pane scrollback.
3. **Agent launcher / CLI host** — spawn the user's login shell (preserving `.zshrc`/`.bashrc`/PATH) and drop into it or send an agent command; one-click launch for Claude Code / Codex / Gemini / Aider / OpenCode. **BYO auth — we never store or broker provider credentials.**
4. **Workspaces** — a workspace = project directory + pane layout; multiple color-coded workspace tabs; `yourapp .` launcher from any shell.
5. **Persistence & restore** — persist workspaces, layout tree, per-pane cwd + last agent/command in **SQLite**; restore layout on relaunch. Lean on each agent's own resume (`claude --resume`, `codex resume`) rather than freezing live processes.
6. **Persistent PTY-host process** — PTYs are owned by a backend process, **not** the window, so a renderer/UI crash never kills a running agent; the UI reconnects. (This is the single most impactful reliability decision.)
7. **Agent-state detection via OSC** — parse the PTY stream for **OSC 9 / 99 / 777** (attention/notifications), **OSC 133** (command start/end + exit code), **OSC 7** (cwd). Surface as tab **rings/badges** and **busy/idle** per pane.
8. **Warp-style command blocks** — collapsible per-command sections with exit-code color, timestamps, search.
9. **Per-pane git awareness** — branch + dirty state in a sidebar (read-only in MVP).
10. **Scriptability seed** — a local socket + `yourapp notify` helper (wire into agent hooks) and minimal `send`/`send-key`.
11. **Distribution basics** — code signing + notarization, signed auto-update. (A desktop app users won't install if it's unsigned.)

**Later (v1+ — differentiators, in rough priority order):**
- **Git worktree-per-agent isolation** + **pre-ship diff review** (secret-redacting, injection-resistant — match BridgeSpace).
- **Kanban board** that launches an agent into a pane with task context.
- **Control API** to tmux/cmux parity (`list`, `send`, `send-key`, `capture`, layout ops).
- **Multi-agent orchestration** ("swarm": roles, exclusive file ownership, shared mailbox, reviewer gate).
- **MCP-shared persistent memory** (a `.memory/` markdown graph).
- **Integrated editor + file tree**, **SSH/remote runtimes**, **multi-profile account switching + usage-limit failover** (orchestrating *which CLI profile is active* — still never brokering auth).
- **Linux**, voice, built-in browser.

## 5. Recommended tech stack & architecture

### Decision: **Electron + xterm.js (WebGL addon) + node-pty**, with a persistent PTY-host process and a platform-agnostic TS "engine" package.

**Why Electron over Tauri (the load-bearing call):**
A terminal multiplexer lives or dies on **consistent, high-performance rendering of many streaming PTYs across both OSes**. That is exactly where the engine choice matters:
- **Electron ships one Chromium engine on Windows and macOS.** You tune the WebGL terminal renderer **once** and it behaves identically on both. **Tauri uses the *system* WebView — Chromium-based WebView2 on Windows vs. WebKit WKWebView on macOS** — so you get two divergent engines, two sets of rendering/perf/clipboard/font quirks, and you chase bugs twice. **This is almost certainly the root of BridgeSpace's documented multi-month terminal-rendering/freeze history.** Since our explicit wedge is *parity + rendering reliability*, Electron removes the single biggest source of parity bugs.
- **The stack is the most battle-tested terminal-in-a-webview combination in existence:** VS Code's integrated terminal is **Electron + xterm.js + node-pty**. xterm.js is maintained by the VS Code team; node-pty is maintained by Microsoft and wraps **ConPTY on Windows and forkpty on Unix behind one JS API**. You inherit a decade of hardening for free.
- **node-pty is turnkey for the core requirement** ("real PTY on Windows + macOS from one codebase"). In Tauri you'd instead wire Rust's `portable-pty` (wezterm) over IPC to the WebView — viable, but more plumbing for the same result, on top of the WebView-divergence tax.
- **Ecosystem & hiring:** far more terminal/PTY prior art (Hyper, VS Code, Theia) and JS talent.

**Honest tradeoffs / when Tauri would win:** Electron costs ~100–150 MB binaries and higher RAM; Tauri is far leaner and "more native." If minimal footprint were the top priority (it isn't — this is a heavyweight multi-agent dev tool) or if you wanted to match BridgeSpace's stack, Tauri + `portable-pty` + xterm.js is a real option. But for a small team whose mandate is *Windows/macOS parity and rendering reliability*, Electron is the faster, safer path. **Hedge:** keep the core (PTY host, OSC parser, session store, orchestration) as a **platform-agnostic TypeScript package decoupled from Electron**, so it's unit-testable headless, could power a CLI/headless mode, and could be re-hosted in Tauri or a native shell later without a rewrite.

### Architecture (three tiers)

**1. Renderer (Chromium window) — UI only.**
- **xterm.js** per pane with **`@xterm/addon-webgl`** (GPU), plus `addon-fit`, `addon-search`, `addon-unicode11`, `addon-serialize` (scrollback snapshotting), `addon-web-links`, clipboard. Canvas renderer as fallback.
- Layout tree (splits/tabs) in React/Svelte; a light state store. **Security hardening:** `contextIsolation: true`, `nodeIntegration: false`, `sandbox: true`, no `@electron/remote`; all privileged ops over typed IPC.

**2. PTY-host / engine process (Node, `utilityProcess` or a separate daemon) — owns everything stateful.**
- **PTY spawning (node-pty):** one PTY per pane. Spawn the **login shell** (`$SHELL -l` on macOS; `pwsh`/`powershell`/`cmd` on Windows) so user config + PATH load, then optionally auto-send the agent command. This preserves TUI behavior (alt-screen, raw mode, 256/truecolor) that agent CLIs need.
- **Process lifecycle:** track pid, cwd, exit code; forward resize (`pty.resize` → SIGWINCH); **kill the process tree** on pane close (Windows: job objects / `taskkill /T`; Unix: process-group kill); reap finished background agents.
- **Windows/ConPTY specifics to plan for:** node-pty uses ConPTY (Win10 1809+) with a winpty fallback; watch resize races, signal semantics, and the occasional TUI that misbehaves under ConPTY; expect **antivirus/SmartScreen** friction on spawned children and on the unsigned dev build — sign early.
- **OSC parser (in this process, on the raw PTY stream):** detect **OSC 9 / 99 / 777** → "needs attention" (tab ring/badge); **OSC 133 A/B/C/D** → command boundaries + exit code (drives command blocks and busy↔idle); **OSC 7** → cwd tracking; ignore/pass-through the rest. Because not every CLI emits OSC, add a **fallback heuristic** (output quiescence after input + alt-screen state) for idle/busy.
- **First-party hook channel (deep integrations):** for Claude Code/Codex, optionally install lightweight hooks (e.g., `SessionStart`/`Stop`/`Notification`) that call a bundled **`yourapp notify`** helper over a **local socket / named pipe** — the same pattern cmux uses. Combine "OSC works for *any* CLI" with "hooks give *reliable* signals for the CLIs we integrate deeply." **Never** put provider credentials in these hooks.
- **Persistence (SQLite via `better-sqlite3`):** workspaces, layout tree (JSON), per-pane cwd + agent command, command-block history, optional serialized scrollback. Matches BridgeSpace's SQLite choice.
- **Session model / "resume":** live PTYs die when their host dies, so keep the PTY-host **separate from the window** → UI crash ≠ agent death. For full app/OS restart, **restore layout + cwd and relaunch agents using their own `--resume`/`resume` flags** rather than trying to freeze processes (cheaper and correct). A true tmux-style always-on daemon that survives full quit is a *later* option, not MVP.

**3. Main process — window/lifecycle, IPC broker, updater, tray, deep-link (`yourapp://`, `yourapp .`).**

### Build/distribution gotchas to budget for
- **Native modules:** node-pty and better-sqlite3 are native → `electron-rebuild`/prebuilds for **win-x64, mac-arm64, mac-x64** (and universal mac if desired).
- **Packaging & signing:** `electron-builder`; **Apple notarization + hardened runtime** (macOS) and **Authenticode signing** (Windows) are mandatory for install-without-warnings; **`electron-updater`** for signed background auto-update.
- **Crash/telemetry:** Sentry from Phase 1 (BridgeSpace uses it) — you *will* need it to chase rendering/PTY edge cases.
- **Perf budget:** define a target (e.g., 16 panes streaming at 60 FPS, capped RAM); throttle/virtualize background panes; cap scrollback.

## 6. Open-source repos to study or fork, grouped by purpose (with license flags)

**Copyleft warning up front:** **AGPL-3.0** (coder/mux, claude-squad) and **GPL-3.0** (cmux) are strong copyleft — you **cannot** lawfully fork/link their code into a closed or source-available commercial product without releasing your source under the same terms (AGPL even triggers on *network* use). Treat these as **study-only / clean-room**: learn the ideas, write your own code. **Permissive** (MIT/Apache-2.0/ISC/BSD) repos are safe to build on.

### A. Terminal + PTY stack — build directly on these (all permissive)
- **xterm.js** — https://github.com/xtermjs/xterm.js — **MIT**. Your terminal frontend; use the WebGL + fit + search + serialize addons. **Use directly.**
- **node-pty** — https://github.com/microsoft/node-pty — **MIT**. ConPTY/forkpty behind one API. **Use directly.**
- **microsoft/terminal** — https://github.com/microsoft/terminal — **MIT**. Reference for ConPTY behavior/edge cases. **Study.**
- **wezterm `portable-pty`** — https://github.com/wezterm/wezterm — **MIT**. Only if you ever take the Tauri/Rust path. **Reference/optional dep.**
- **ghostty / libghostty** — https://github.com/ghostty-org/ghostty — **MIT**. The native GPU engine cmux embeds; reference if you ever want a native renderer. **Study.**

### B. Direct competitors — study the UX/protocols, do NOT fork the copyleft ones
- **manaflow-ai/cmux** — https://github.com/manaflow-ai/cmux — **GPL-3.0 + commercial dual license** (GitHub also reports NOASSERTION — terms not fully pinned). The reference implementation for **OSC 9/99/777 notification rings** and a **socket + `cmux send/send-key` API**. **Study only; clean-room the ideas.**
- **coder/mux** — https://github.com/coder/mux — **AGPL-3.0**. Study the **Local/Worktree/SSH runtime model** and git-divergence view. **Study only (AGPL).**
- **smtg-ai/claude-squad** — https://github.com/smtg-ai/claude-squad — **AGPL-3.0**. Canonical **tmux + git-worktree per-agent** session pattern. **Study only (AGPL).**
- **Conductor** (conductor.build, Melty Labs) — closed source, no repo. **UX benchmark only** for the desktop parallel-agents experience.

### C. Orchestration / worktree engines — permissive, you MAY reuse or fork
- **BloopAI/vibe-kanban** — https://github.com/BloopAI/vibe-kanban — **Apache-2.0** (permissive). ~27k stars; the closest analog to Kanban-over-agent-CLIs with **worktree-per-task** launching. **Fork/reuse candidate** for the v1 Kanban + worktree layer.
- **devflowinc/uzi** — https://github.com/devflowinc/uzi — **MIT** (permissive; but **stale**, last push mid-2025). A headless **worktree + tmux swarm** engine. **Reuse ideas/code**, verify freshness.
- **imbue-ai/sculptor** — https://github.com/imbue-ai/sculptor — **MIT** (README caveats "experimental / not truly OSS yet"). Container-isolated parallel agents + "Pairing Mode." **Study; reuse cautiously.**

### D. Reference terminals / multiplexers (permissive) — study patterns
- **tmux** — https://github.com/tmux/tmux — **ISC**. The model for **daemon-backed session persistence** and a **control-mode/`send-keys` scripting API**. **Study** (the scripting UX we should emulate).
- **Hyper** — https://github.com/vercel/hyper — **MIT**. Electron + xterm.js reference architecture. **Study/borrow.**
- **VS Code integrated terminal** (in microsoft/vscode) — **MIT**. The gold reference for **xterm.js + node-pty in Electron** at scale. **Study.**

### E. Later — memory/MCP (permissive)
- **modelcontextprotocol** SDK/servers — https://github.com/modelcontextprotocol — **MIT**. For the v1+ shared-memory/knowledge-graph feature.

**Net licensing guidance:** build the product on **B/A/C-permissive** (xterm.js, node-pty, vibe-kanban, uzi, Hyper, tmux ideas); keep **cmux, coder/mux, claude-squad** strictly at arm's length (clean-room). Decide your own license (permissive source-available like BSL/Elastic-style, or closed) — a non-AGPL posture is itself a competitive selling point.

## 7. Build phases / rough roadmap

- **Phase 0 — Parity spike (1–2 wks).** Electron + xterm.js(WebGL) + node-pty. One live PTY pane rendering an actual **Claude Code** session **on both Windows (ConPTY) and macOS (forkpty)**. Prove identical rendering + input + resize. *Gate: if rendering diverges, revisit engine choice now, not later.*
- **Phase 1 — MVP core (4–8 wks).** Multi-pane grid + workspace tabs; **persistent PTY-host process + reconnect-after-UI-crash**; SQLite persistence + layout/cwd restore; shell/PATH fidelity; agent launcher for the CLI roster; signing/notarization + auto-update wired in.
- **Phase 2 — Agent awareness (3–5 wks).** OSC parser (9/99/777/133/7) → tab rings/badges + busy/idle; **Warp-style command blocks** + exit codes + search; per-pane git branch/dirty; `yourapp notify` socket + first-party Claude/Codex hooks. *Milestone demo: "16 agents, see at a glance who needs you, nothing freezes."*

- **Phase 4 — Differentiators (ongoing).** Multi-agent **swarm** (roles/file-ownership/mailbox/reviewer gate); **MCP shared memory**; integrated editor/file tree; SSH/remote runtimes; multi-profile + usage-limit failover; Linux; voice/browser.

## 8. Open questions & risks

1. **Rendering perf at high pane counts (the make-or-break risk).** 16 agents streaming simultaneously must not freeze — this is BridgeSpace's exact failure mode. Needs a perf budget, background-pane throttling/virtualization, scrollback caps, early load testing. *Electron de-risks the divergence axis but not raw perf.*
2. **True persistence across full app/OS restart.** PTYs die with their host. MVP leans on agents' own `--resume`; a real tmux-style survive-quit **daemon** is a bigger build — decide if/when it's worth it.
3. **Windows/ConPTY quirks.** Resize races, signal semantics, misbehaving TUIs, winpty fallback, and **AV/SmartScreen** friction on spawned processes + unsigned builds. Budget real time here; it's where "cross-platform" projects bleed.
4. **BYO-auth boundary (legal + trust).** We must **never** store/broker provider credentials. "Multi-profile account switching" must mean *orchestrating which CLI profile/env is active*, not holding provider secrets — keep this crisp in design and messaging.
5. **Agent-CLI churn.** Claude Code/Codex/Gemini change flags, hooks, and output formats frequently; deep hook integrations are brittle and a standing maintenance cost. Prefer OSC (works for any CLI) as the primary channel, hooks as an enhancement.
6. **Not every CLI emits OSC.** Idle/busy fallback heuristics will have false positives/negatives; set expectations and let users override.
7. **Licensing hygiene.** Zero code from AGPL/GPL rivals (cmux, coder/mux, claude-squad) — clean-room only. Choose our own license deliberately (non-AGPL is a selling point).
8. **Security surface.** We host arbitrary CLIs that execute code and read repos; the pre-ship diff review must redact secrets and resist prompt injection (as BridgeSpace does). We inherit that surface the moment we add diff review.
9. **Competitive/positioning risk — Warp.** Warp is well-funded, already cross-platform (incl. Windows), adding MCP + multi-agent management, and re-branding as an ADE. Betting on *neutral organizer + Windows parity + reliability + scriptability* must be executed sharply; "another Warp" loses.
10. **Business/IP.** "Custom version of BridgeSpace" — ensure clean-room engineering, no BridgeMind IP/branding/asset reuse, and clear our own name on trademark (see §9).
11. **Scope discipline.** BridgeSpace's surface is enormous (editor, browser, voice, memory, swarm). The failure pattern is chasing all of it; **nail the organizer core (§4 must-haves) first.**

## 9. Naming foundation — 12 directions (seed for a fuller naming exercise)

Grounded in the identity: *one calm control surface that runs, arranges, and coordinates many parallel AI-coding agents across Windows and macOS.* **All candidates below are directions, not final picks — each needs a trademark + domain (.com/.dev/.ai) + collision check.** Avoid known collisions in this space: *Warp, tmux, cmux, coder/mux, Conductor, Ghostty, Hyper, Zellij, WezTerm, WaveTerm/Wave, Tabby, Kitty, Alacritty, claude-squad.*

### One-paragraph bottom line
Build an **Electron + xterm.js + node-pty** desktop app whose backbone is a **persistent PTY-host process** delivering **identical, reliable terminal rendering on Windows (ConPTY) and macOS (forkpty)** — then win by being the **neutral, scriptable, rock-solid organizer** of first-party agent CLIs (never brokering auth) exactly where cmux/Conductor skip Windows and BridgeSpace fights rendering bugs. ## ANGLE: Framework choice for a cross-platform (Windows + macOS) terminal app with workspaces and BYOK AI, feeding a BridgeMind/Warp/tmux competitor. Each viable stack pairs a UI layer with a PTY layer, and the three differ sharply on terminal/PTY maturity.

TAURI (Rust backend + OS WebView + xterm.js frontend + portable-pty PTY crate): smallest/fastest of the three — bundles ~2-10 MB, idle RAM ~30-50 MB, ~4x faster startup than Electron (tech-insider.org, pkgpulse.com, gethopp.app 2026 benchmarks). This is exactly the stack BridgeMind's BridgeSpace uses (Tauri + xterm.js/WebView2 + portable-pty). BUT terminal/PTY maturity is the weakest: the building blocks are solid (portable-pty — wezterm's cross-platform PTY abstraction incl. Windows ConPTY, v0.9.0; plus tauri-plugin-pty), yet there is NO large flagship production terminal shipping on Tauri. Every Tauri terminal repo I found is small/hobby-scale (largest = marc2332/tauri-terminal ~125 stars, a 2023 demo). Key caveat: per-platform WebView divergence (Chromium/WebView2 on Windows vs WebKit on macOS) causes font/CSS rendering differences, and BridgeSpace's own changelog shows a multi-month history of xterm/WebView freeze bugs — consistent with Tauri-terminal immaturity.

ELECTRON (Chromium + Node + xterm.js + node-pty): heaviest — bundles ~80-200 MB, RAM ~120-300 MB, slower cold start — but by far the most mature and battle-tested for terminals. xterm.js and microsoft/node-pty were built for VS Code's integrated terminal. Every major 'terminal-with-superpowers' ships on Electron: VS Code (186k*), Tabby (72k*), Hyper (44k*), Wave Terminal (21k*, Electron UI + Go backend, AI-integrated with BYOK across OpenAI/Claude/Gemini/Ollama — the closest existing analog to the target product), electerm (14k*).

FLUTTER DESKTOP (Dart + Skia/Impeller GPU): middle ground on bundle size, native GPU rendering, RAM lower than Electron. Terminal/PTY maturity is niche but real, carried almost entirely by one maintainer (TerminalStudio): xterm.dart widget (651*, 60fps, CJK/emoji) + flutter_pty (30*). macOS is Flutter's most mature desktop target; Windows/Linux have fewer production-grade packages. IMPORTANT: I found NO notable, widely-used production terminal app built on Flutter desktop, and xterm.dart's last commit was mid-2025 (less active than the others) — stating that gap explicitly rather than guessing.

NATIVE RUST (not a cross-platform app framework, but the performance ceiling the competitors set): Warp itself is native Rust + a custom wgpu GPU renderer (Metal/Vulkan/DX12), >144 fps, ~1.9 ms redraw — no Electron/WebView; new in 2026 its client codebase is open source (warpdotdev/warp, 62k*, dual-licensed MIT for the warpui UI crates + AGPL-3.0 for the rest, Rust 98%). WezTerm (27k*, native Rust) is the origin of the portable-pty crate. Rio (7k*) is another Rust/GPU terminal. These set the speed bar but require building your own UI toolkit.

BOTTOM LINE: Electron = fastest path to a robust workspace terminal with the richest PTY ecosystem (node-pty) and the most reference apps to copy (Wave/Tabby/Hyper). Tauri = smallest/snappiest and matches BridgeSpace, but you'd be pioneering terminal-specific PTY/WebView stability at app scale (the portable-pty crate is production-grade; the app-level Tauri-terminal ecosystem is thin). Flutter = only compelling if you already have Dart/Flutter skills and lead with macOS; thinnest terminal ecosystem and no flagship precedent. ## ANGLE: Open-source unified multi-provider LLM SDKs and self-hostable gateways that enable bring-your-own-key (BYOK) across Anthropic Claude, OpenAI, and others — evaluated for embedding into a cross-platform BridgeMind/BridgeSpace-style desktop terminal app (Tauri/WebView2, so TypeScript- and native-sidecar-friendly) that lets users plug in their own AI subscription keys. Two architectural patterns emerged: (1) in-process SDKs/libraries that normalize provider APIs in code (Vercel AI SDK, aisuite, simonw/llm), and (2) standalone gateway/proxy servers you can bundle as a local sidecar that expose one OpenAI-compatible endpoint and fan out to many providers (LiteLLM, Portkey Gateway, Bifrost, one-api, LLM Gateway). All abstract provider differences by converging on the OpenAI Chat Completions schema (or a provider:model string) and translating requests/responses; keys are supplied either via env vars, a per-provider factory/config, or a key-vault/channel layer. NOTE: I did NOT include OpenRouter itself as a repo because OpenRouter's core routing service is a hosted product, not an open-source repo — but it is highly relevant as a reference BYOK model: it exposes an OpenAI-compatible API, supports OAuth + BYOK (you register provider keys and it uses them, first 1M BYOK requests/mo free then a 5% fee), and the community 'awesome-openrouter' list and OpenRouter SDKs (TS/Python/Go/Java) are the integration surface. All star counts below are from the GitHub page/README at fetch time (late June/July 2026) and will drift. Every URL was confirmed via search + direct fetch; none are guessed.

    relevance: Most directly embeddable in a Tauri/WebView2 desktop terminal app since that stack is TS/JS-native — you can call Claude and OpenAI from the app's own frontend/Node layer without shipping a separate server. Abstracts provider differences by normalizing to one message/tool/streaming schema so switching providers is ~2 lines. BYOK is handled per-provider via factory functions (e.g. createAnthropic({apiKey}) / createOpenAI({apiKey})) or env vars (ANTHROPIC_API_KEY, OPENAI_API_KEY) — a natural fit for a Settings 'API Keys' tab where users paste their own subscription keys.
